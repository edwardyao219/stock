# 第一轮：收盘数据同步与完整性门禁设计

## 目标

第一轮只解决三个问题：

1. 让 15:30 定时任务真实执行全市场收盘数据同步。
2. 当日数据不完整时，阻断次日新候选、新计划和新开仓所依赖的数据批次。
3. 在 Web 中明确区分实时指数、最近交易日宽度和低覆盖降级数据。

本轮不新增策略因子、不调整市场状态阈值、不引入新的任务编排或告警系统。

## 现状与根因

- `services.jobs.tasks.sync_daily_market_data_task` 当前只返回 `pending`，15:30 并未执行同步。
- 18:00 盘后流程会尝试全市场同步，但这是候选发现前的单一兜底点。
- `/market/overview` 已返回日期、覆盖率和快照范围，但 Web 的主摘要仍直接显示涨跌家数，旧数据不够醒目。
- 现有特征覆盖率用于衡量可扫描股票，不等同于当日日线是否完整。两类覆盖率必须分开判断。

## 方案选择

### 采用方案

复用现有同步器、盘后流水线和任务状态结构，补一个共享的日线完整性门禁。15:30 定时任务和 18:00 盘后流程使用同一同步入口及同一门禁语义。

### 未采用方案

- 不只修改 15:30 任务而忽略盘后流程，否则手工运行和定时运行可能产生不同结论。
- 不新建独立编排服务或事件总线，当前任务规模不需要这类基础设施。
- 不将候选特征覆盖率强行提高到 98%。特征会因历史长度等原因少于日线，日线完整性和特征可扫描性继续使用不同阈值。

## 数据流

```text
15:30 定时任务
  -> 全市场日线及配套数据同步
  -> 记录各数据集结果

18:00 盘后流程
  -> 再次同步/补齐当日数据
  -> 计算当日日线完整性
  -> 完整：计算特征 -> 发现候选 -> 生成计划 -> 纸面流程
  -> 不完整：记录阻断原因 -> 跳过新候选和新计划
```

已有持仓的盘中退出监控不依赖新候选批次，数据门禁不得关闭退出和风控处理。

## 完整性门禁

门禁以目标交易日为准，动态使用当前有效、非 ST 的 A 股证券数量作为预期分母，不写死股票总数。

门禁通过需要同时满足：

- 数据日期等于目标交易日。
- 当日日线数 / 预期证券数不低于 98%。
- 当日成交额缺失率低于 1%。

预期证券数为零或目标交易日没有日线时，门禁直接不通过。

输出至少包括：目标日期、实际日线数、预期证券数、覆盖率、成交额缺失率、是否允许生成候选，以及中文阻断原因。

特征覆盖率继续作为独立指标。日线门禁通过后，特征覆盖不足仍沿用现有候选降级逻辑，不与 98% 日线阈值混用。

## 阻断行为

当门禁不通过时：

- 不运行当日的新候选发现。
- 不生成依赖该批次的新交易计划。
- 盘后状态标记为 `warning`，并展示覆盖率及阻断原因；只有同步代码异常或任务无法执行才标记为 `failed`。
- 不删除上一批候选，也不把旧候选伪装成当日结果。
- 已有持仓的退出、止损和纸面风控继续运行。

## 前端展示

主摘要中的市场宽度、成交额和市场压力必须显示其数据范围：

- 当日实时数据使用“盘中实时”。
- 落库收盘数据使用“最近交易日”，并显示具体日期。
- 覆盖不足时显示“覆盖不足”和实际样本数/预期样本数。
- 实时指数可以继续展示，但其日期单独显示，不能让用户误以为宽度和指数来自同一时点。

本轮复用现有摘要区域，不新增页面、弹窗或复杂组件。

## 错误处理

- 同步异常保留具体数据集、异常类型和行数，避免只返回笼统的 `pending`。
- 15:30 同步失败后，18:00 盘后流程仍会再次同步，因此本轮不新增独立重试服务。
- 数据源部分成功时，最终是否允许候选生成只由完整性门禁决定，不能因为某个接口返回 `ok` 就默认数据可用。
- 非交易日沿用交易日历保护，不生成空的当日批次。

## 测试与验收

采用测试先行，至少验证：

1. 15:30 任务调用真实的全量同步入口，不再固定返回 `pending`。
2. 同步结果为部分成功时，完整性门禁仍按数据库实际覆盖判断。
3. 日线覆盖率低于 98% 时，盘后流程不会调用候选发现。
4. 成交额缺失率达到 1% 时，盘后流程不会调用候选发现。
5. 门禁失败不会改变盘中已有持仓退出监控的执行路径。
6. Web 对旧日期和低覆盖市场数据显示明确中文标识。

回归范围包括任务流水线、市场 API、候选发现、纸面实盘相关测试、前端标签测试和前端构建。

## 完成标准

- 定时同步任务返回真实同步结果。
- 盘后状态能说明当日数据是否允许生成候选及具体原因。
- 不完整数据无法产生新的当日候选批次。
- Web 不会把最近交易日宽度展示成今日实时宽度。
- 相关后端测试、前端测试、静态检查和构建通过。
