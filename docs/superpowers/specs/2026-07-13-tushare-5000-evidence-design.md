# Tushare 5000 积分数据证据链设计

## 目标

将已验证可用的 Tushare 5000 积分接口转成系统可追溯的辅助证据。本轮接入三个每日数据集：

- `moneyflow_dc`：东财个股资金流。
- `limit_list_d`：涨跌停与开板结构。
- `cyq_perf`：筹码成本与获利盘分布。

数据进入候选解释、交易计划证据和盘后诊断。明确风险可以降低行动级别；正向证据不能单独提高核心候选等级。

本轮不接入 `moneyflow_ths`，避免与现有个股资金流和东财资金流重复计分；`ths_index`、`ths_member` 留到独立的板块成分时点化项目。

## 已验证权限

使用 2026-07-10 只读探测确认当前代理权限可用：

| 接口 | 返回行数 | 用途 |
|---|---:|---|
| `limit_list_d` | 187 | 涨跌停、开板、连板结构 |
| `moneyflow_dc` | 5907 | 个股净流入与大单方向 |
| `moneyflow_ths` | 5195 | 暂不接入，避免重复资金因子 |
| `cyq_perf` | 5521 | 成本分位与获利盘比例 |
| `ths_index` | 409 | 下一轮板块成分快照基础 |

权限探测只记录接口、字段和行数，不输出 token、代理地址或认证内容。

## 数据模型

新增三张原始表，均保留 Tushare 原始单位，不在采集层改变数值含义。

### `tushare_moneyflow_dc`

以 `(ts_code, trade_date)` 唯一，保存：

- `name`、`pct_change`、`close`
- `net_amount`、`net_amount_rate`
- 超大单、大单、中单、小单金额及比例

### `tushare_limit_list_d`

以 `(ts_code, trade_date)` 唯一，保存：

- `industry`、`name`、`close`、`pct_chg`、`amount`
- `limit_amount`、`fd_amount`、`turnover_ratio`
- `first_time`、`last_time`、`open_times`
- `up_stat`、`limit_times`、`limit`

### `tushare_cyq_perf`

以 `(ts_code, trade_date)` 唯一，保存：

- `his_low`、`his_high`
- `cost_5pct`、`cost_15pct`、`cost_50pct`、`cost_85pct`、`cost_95pct`
- `weight_avg`、`winner_rate`

三个同步器使用现有批量 upsert 和唯一约束，重复执行不得产生重复记录。

## 同步链路

三个数据集加入现有 Tushare 可恢复同步注册表和盘后数据集列表。

```text
15:30 定时同步
  -> 基础日线/每日指标/涨跌停价/个股资金流
  -> moneyflow_dc/limit_list_d/cyq_perf

18:00 盘后流程
  -> 调用同一完整同步入口再次补齐
  -> 数据健康检查
  -> 特征、候选、计划与复盘
```

18:00 不再只补基础日线。全量同步通过后再计算市场特征，避免增量证据停留在上一交易日。

每个数据集独立返回 `ok`、`skipped` 或 `failed` 和实际行数。三个增量数据集失败不会使基础日线门禁自动失败，但候选证据必须显示缺失，盘后状态必须记录失败的数据集。

## 日期与未来函数

- 候选、计划和回放只能查询 `trade_date == signal_date` 的记录。
- 当日数据缺失时返回空证据，不查询“最近一条”旧记录代替。
- 后续回放只使用历史上已落库或按日期重新获取的记录。
- `ths_member` 在完成按日期归档前不得参与历史板块判断。

## 证据转换

原始数据在计划上下文层批量加载，避免逐股查询。转换后只输出低维、可解释字段。

### 东财资金确认

- `net_amount_rate >= 1%`：东财资金方向偏正。
- `net_amount_rate <= -1%`：东财资金方向偏负。
- 与现有 `moneyflow_support_score >= 54` 同时偏正：记录“双源资金确认”。
- 与现有 `moneyflow_support_score <= 45` 同时偏负：记录“双源资金流出风险”。
- 两个来源方向相反：记录“资金来源分歧”，不得作为升级依据。

东财资金流不生成新的独立总分，避免同一资金信息被重复加权。

### 涨跌停结构

- `limit == "D"`：记录跌停风险。
- `limit == "U"` 且 `open_times >= 2`：记录反复开板风险。
- 首次封板、连板次数和封板时间只写入解释，不单独提高候选等级。
- 非涨跌停股票没有记录属于正常情况，不标记为数据缺失。

### 筹码位置

- 收盘价低于 `cost_50pct`：记录上方筹码压力。
- `winner_rate >= 90%` 且收盘价不低于 `cost_85pct`：记录高获利盘过热风险。
- 其他状态只生成中性筹码描述，不产生升级。

阈值固定为初始解释规则。任何调整必须经过滚动样本外验证，不能根据单月表现修改。

## 候选与计划行为

- 数据先进入计划上下文和证据列表。
- “双源资金流出风险”、跌停、反复开板和筹码过热可以阻止行动级计划，候选保留在观察层。
- “双源资金确认”、首次封板和健康筹码结构只能增强解释，不改变核心候选数量上限和市场状态门控。
- 缺少任一增量数据集时保持原有策略结果，同时显示具体证据缺失，不把缺失当作负面。

## 盘后状态

盘后状态新增 Tushare 数据集摘要：

- 数据集交易日、行数和状态。
- `moneyflow_dc` 与 `cyq_perf` 相对当日日线股票的匹配覆盖率。
- `limit_list_d` 事件行数，不使用全市场覆盖率衡量。
- 失败数据集和候选证据降级说明。

Web 复用现有收盘状态抽屉显示这些信息，不新增独立页面。

## 错误处理

- 单个接口失败时继续同步其他数据集。
- 强制重跑使用 upsert 修复部分数据。
- 可选数值字段缺失时保存为 `None`；缺少 `trade_date`、`ts_code` 等唯一键或响应无法解析时，整个数据集标记为失败，不写入残缺主键记录。
- 数值为 `None` 时不参与证据判断。
- token、密码和代理认证不得进入日志、状态、测试或提交。

## 测试与验收

采用测试先行，覆盖：

1. 三个接口字段解析和批量 upsert。
2. 相同交易日重复同步不产生重复记录。
3. 三个数据集进入可恢复同步注册表，并由 15:30、18:00 共用。
4. 候选上下文按信号日批量加载，缺失时不回退旧日期。
5. 资金一致、资金分歧、跌停、反复开板、筹码压力和筹码过热证据。
6. 正向证据不能独立升级候选，明确风险可以阻断行动计划。
7. 盘后状态包含数据集行数、覆盖率和失败原因。
8. 原有日线门禁、候选、纸面实盘、回放和前端构建回归通过。

## 完成标准

- 2026-07-10 三个数据集可通过现有同步入口幂等落库。
- 15:30 和 18:00 均能补齐三个数据集。
- 候选和计划能显示同日资金、涨跌停和筹码证据。
- 明确风险能降级行动计划，正向证据不会绕过市场和板块门控。
- 盘后状态能说明数据是否到齐以及缺失对候选的影响。
- 无未来函数、无 token 泄漏，相关测试、Ruff 和前端构建通过。
